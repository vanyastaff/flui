# FLUI Port Roadmap

> The dependency-ordered construction plan for porting Flutter to Rust ‚Äî from the current codebase to full Flutter parity. It is written **backward from the finished product**: released Flutter is the specification, full parity is the destination, and the phases are the dependency-correct path to it. Progress is measured as **parity against `.flutter/`**, not as crates touched.

This roadmap sits on top of [`FOUNDATIONS.md`](FOUNDATIONS.md) ‚Äî the architecture contract. The foundations say *what* is built and *to what rules*; this document says *in what order*. Phases are ordered purely by dependency correctness and risk. There are **no calendar dates** ‚Äî a phase is done when its exit criteria are objectively met.

## The destination

The target is **full parity with released Flutter** ‚Äî every framework package, adapted to Rust-native structure and improved where Rust permits ([`FOUNDATIONS.md` Part II](FOUNDATIONS.md#part-ii--where-flui-is-better-than-flutter)). Released Flutter is ~480k LOC of framework logic across 12 packages. FLUI today covers an estimated **~35‚Äì40% of Flutter's framework logic** (a coverage-weighted estimate of what is built and working ‚Äî not raw LOC √∑ raw LOC; the raw FLUI workspace is ~292k Rust code LOC in `crates/` as of 2026-07-14, ~295k including `examples/` and `tools/`, but that includes tests and scaffolding). The estimate is sharply bimodal in a new place: the render *machine*, render-object *catalog*, and framework widget *catalog* are largely built; the design-system layer (`material`/`cupertino`) is ~0%.

## What parity means and how it is measured

"Full parity with released Flutter" is the destination. To prevent that destination from being an undefined "we're done when it feels done," parity is given a **falsifiable definition** and an **acceptance oracle** ‚Äî without these "one march to 100%" has no forcing function and "no intermediate release" means no release ever.

**Definition.** A FLUI widget, render object, or subsystem is at **parity** with its Flutter counterpart when:
1. It implements the same behavior contract ‚Äî the same `.flutter/` algorithm, lifecycle, and observable semantics (per [`FOUNDATIONS.md` Part I](FOUNDATIONS.md#part-i--the-target-architecture) "behavior loyal").
2. It passes the **adapted Flutter widget-test corpus** for that component (the oracle below).
3. It satisfies the FLUI-side quality bar: the coverage threshold for its crate category (Core ‚â• 80%, Platform ‚â• 70%, Widget ‚â• 85% ‚Äî checked locally via `just coverage`; there is currently **no CI coverage gate**), all `port-check.sh` refusal triggers green (22 numbered triggers plus the named guards as of 2026-07-14), no `unimplemented!()`/`todo!()` in its code path.

**The oracle.** Flutter ships an enormous widget-test corpus under `.flutter/flutter-master/packages/flutter/test/`. Each widget has a `_test.dart` companion exercising layout, paint, gestures, edge cases, accessibility. The parity oracle for FLUI is: **the corresponding Flutter `_test.dart` is ported (mechanically where possible, behavior-faithfully always) and passes against the FLUI widget**. This is the same "behavior loyal, structure Rust-native" rule applied to tests. The ported corpus lives at `crates/<crate>/tests/parity/` and runs in CI ‚Äî it is live today in `crates/flui-widgets/tests/parity/`.

**Measurement.** The parity scoreboard at the top of this document reports a **coverage estimate** ‚Äî what is built and working, weighted by component importance. With the ported test corpus online, the scoreboard's second dimension is **fidelity** = fraction of ported Flutter tests passing. A package is at full parity only when **both** reach 100% ‚Äî coverage answers "how much is built," fidelity answers "and does it behave like Flutter." Coverage alone is not done. **Fidelity is now the roadmap's main open front for `widgets`:** coverage ran ahead of the ported corpus.

**Why this matters for "one march to 100%."** Without a definition, "100%" is unreachable ‚Äî the last 5% is unbounded. With this definition, the last failing ported Flutter test is the gate; when none fail against the corresponding FLUI widget, the package is done. There is a forcing function, and "no intermediate release" no longer means "no forcing function."

- **Cross** ‚Äî the substrate the rest stands on. Foundation hardening, platform backends, animation/physics, asset pipeline, DX tooling (devtools/build/cli/hot-reload), the refusal triggers, the standing audit-driven quality discipline. Not a phase ‚Äî it runs the whole duration in tracks.
- **Core** ‚Äî the render machine. The spine from `View` through `Element` reconciliation through `RenderObject` layout/paint to layered compositing, plus the concrete render-object catalog (`flui-objects`) and the deterministic headless frame driver (`flui-binding`). Includes the contract design docs (the *core* the widget catalog commits to).
- **Business** ‚Äî `flui-widgets`. The ~80-widget framework catalog ‚Äî `Container`/`Row`/`Text`/`Image`/`ListView`/`GestureDetector`/`Navigator`/`Focus`/`AnimatedContainer`/.... The thing an app author composes. Depends on Core + Cross.
- **Catalog** ‚Äî design-system component libraries. `flui-material` ‚à• `flui-cupertino` (independent siblings) + the `flui-localizations` they share. The thing app authors typically import.
- **App** ‚Äî `flui-app` + the `flui` facade. The top-level binding wiring platform vsync ‚Üí frame loop ‚Üí Catalog. The phase that ships a real app.

One crate directory is deliberately **outside** the workspace: `flui-reactivity` (signals/hooks experiment) is excluded from `[workspace.members]` because signals are locked out by contract C1 ‚Äî it re-enters only if signals are ever integrated into the view layer (rationale in the root `Cargo.toml` note).

The phases below sequence construction across these layers. Phase headings use the layer prefix so a reader sees at a glance which layer the work belongs to.

## Construction strategy ‚Äî spine-first vertical slice

The phases follow the dependency graph, with one deliberate shape decision. After the spine was brought to target spec (Core.0), construction did **not** go straight to breadth. Core.1 built a thin **vertical slice** ‚Äî a handful of widgets, one per render-object family, end-to-end ‚Äî *before* the wide render-object and widget catalogs.

The reason is risk. `material` is 210k LOC; an architecture-contract flaw discovered mid-`material` is a catastrophe. The vertical slice exercises every locked contract ([`FOUNDATIONS.md` Part III](FOUNDATIONS.md#part-iii--the-locked-contracts)) and the whole build ‚Üí layout ‚Üí paint ‚Üí composite ‚Üí reconcile pipeline on **live widget code**, cheaply, before the expensive breadth. A paper contract is not proven; a contract a running app depends on is. This strategy has largely played out as designed ‚Äî the slice landed, then the catalogs followed ‚Äî with the combined slice demo app the one piece of it still owed (see Core.1).

## How to read a phase

Each phase states: **Goal**, **Status**, what was **Delivered** (for closed work) and what **Remains** (objective, testable criteria ‚Äî a phase is not done until every item is verifiably true), and **Parity delta** (which `.flutter/` packages it advances). Phases on the critical path are strictly sequential; tracks run in parallel throughout.

## Core.0 ‚Äî Spine to target spec  *(was Phase 0)* ‚Äî ‚úÖ COMPLETE

**Goal.** Bring the render spine up to its target specification ([`FOUNDATIONS.md` Part I](FOUNDATIONS.md#part-i--the-target-architecture)) and lock the architecture contracts the catalog commits to.

**Delivered** (each item verified in code, 2026-07-14):

- **Layout / compositing / paint phases wired to spec.** `run_layout` performs real per-node layout with parent‚Üíchild constraint propagation (`flui-rendering/src/pipeline/owner/layout.rs` ‚Äî it replaced the legacy no-op recursion); `run_compositing` implements the full Flutter subtree compositing-bits walk (`pipeline/owner/compositing.rs`); `run_paint` clears the dirty flag only on nodes it paints (`pipeline/owner/paint.rs`). **Goal.** Prove the locked contracts and the full pipeline on live widget code: ~8‚Äì12 widgets ‚Äî one per render-object family ‚Äî end-to-end, into a running demo app.

- **The parity oracle is live**: `crates/flui-widgets/tests/parity/` (25 files) covers Container, Center, Column/Row, Text, ListView, and the stateful counter.
- **The combined demo app**, `examples/vertical_slice_demo/` ‚Äî a stateful counter, a scrollable list, a gesture-responsive "+" button, and an `AnimatedContainer` box, all in one tree ‚Äî with a shared-tree acceptance test at `tests/vertical_slice_demo.rs` (`#[path]`-includes the example's tree and mounts it headlessly, so the test exercises the exact tree the example runs).

**Parity delta.** `widgets` catalog seeded; `animation` exercised end-to-end; the pipeline is proven on live widget code.

## Core.2 ‚Äî Render-object catalog  *(was Phase 2)* ‚Äî ‚úÖ COMPLETE

**Goal.** Build the ~80-object render catalog. Every widget is a thin configuration object over a render object ‚Äî this was the hidden bottleneck under the widget catalog, and it has been substantially cleared.

**Where it lives.** The catalog is the dedicated **`crates/flui-objects`** crate (extracted out of `flui-rendering` ‚Äî the machine and the catalog are now separate crates). `flui-rendering` keeps only machine types (`RenderTree`, `RenderState`, `RenderView`, `RenderTester`, ‚Ä¶).

**Delivered.** **79 exported render-object types** (74 concrete structs plus type aliases such as the `RenderClipRect`/`RRect`/`Oval`/`Path` family over a generic clip base), enumerated in the `RENDER_OBJECT_TYPES` harness catalog (`crates/flui-objects/tests/render_object_harness.rs`) whose CI guard asserts the list matches the crate's `pub use` exports, with per-type `harness_*` tests. Every family the phase named is implemented:
- **Box layout** ‚Äî `RenderStack`/`RenderIndexedStack`, `RenderConstrainedBox`/`RenderLimitedBox`, `RenderAspectRatio`, `RenderBaseline`, `RenderWrap`, `RenderFlow`, `RenderTable`, `RenderFractionallySizedBox`. (Flutter's `Positioned` is a `ParentDataWidget` over `RenderStack`, not a render object ‚Äî nothing is missing there.)
- **Paint effects** ‚Äî the clip family, `RenderDecoratedBox`, `RenderOpacity` (+ `RenderSliverOpacity`), `RenderAnimatedOpacity` (+ `RenderSliverAnimatedOpacity`), the `RenderTransform` family (+ `RenderFractionalTranslation`, `RenderRotatedBox`), `RenderCustomPaint`, `RenderRepaintBoundary`, `RenderPhysicalModel`/`Shape`.
- **Slivers** ‚Äî `RenderViewport` + `RenderShrinkWrappingViewport`, `RenderSliverList`/`Grid` (each with lazy variants), `Padding`/`FillViewport`/`ToBoxAdapter`, `RenderSliverFixedExtentList`, three `FillRemaining` variants, four persistent-header variants, `Offstage`/`Opacity`/`AnimatedOpacity`/`IgnorePointer`.

- **`RenderAnimatedOpacity`/`RenderSliverAnimatedOpacity`** (the last named gap) ported the mixin's alpha-caching/dirty-marking contract with one **documented divergence**: Flutter's mixin is a retained-layer node ‚Äî a tick calls `updateCompositedLayer` to blend the existing `OpacityLayer` in place, so it never repaints the child subtree, only re-composites. **Exit verification (all met).**

- **Coverage** (`cargo llvm-cov --summary-only -p flui-objects -p flui-rendering`, 2026-07-14): `flui-objects` **81.41%** line coverage, `flui-rendering` **83.27%** line coverage ‚Äî both ‚â• the 80% Core threshold.

## App.1 ‚Äî Application integration  *(was Phase 5)* ‚Äî ‚óê PARTIALLY BUILT

**Goal.** Bring `flui-app` to full parity as the top-level binding integrating the now-complete stack.

**Already in place.** A `runApp`-equivalent exists (`run_app`/`run_app_with_config`/`run_direct`, plus `run_app_android`), `WidgetsFlutterBinding` and `RendererBinding` are implemented, and a working **wake-driven, vsync-paced** frame loop runs today (wake ‚Üí `needs_redraw` ‚Üí `RedrawRequested` ‚Üí begin/draw frame, paced by the GPU's blocking `Fifo` present rather than a fixed sleep) ‚Äî `examples/animated_box_app.rs` runs on it. **True vsync-driven, on-demand pacing** landed 2026-07-17 ([ADR-0029](adr/ADR-0029-frame-pacing-swapchain-block-with-fallback-throttle.md)): `ControlFlow::Wait` is now pinned explicitly in `WinitApp::about_to_wait`, `select_present_mode` defaults to `Fifo` (steady-state pacing = the GPU blocking on `present()` at display cadence), and the runner's old fixed frame-budget sleep is gone, replaced by a coarse fallback throttle that applies only to a frame that never reaches `present()` while a ticker keeps the loop awake. Measured on this machine's real Wayland window (2026-07-17, `xrandr`-confirmed 164.89 Hz display): steady-state inter-tick cadence over 2100 samples across 7 windows sits at median ‚âà 6.058 ms / p90 ‚âà 6.10‚Äì6.15 ms ‚Äî matching the display's own 6.065 ms native period, not a sleep-quantized artifact and not the pre-fix ~30 000 fps busy-spin. **Caveats, stated honestly:** this evidence is Linux/Wayland/Vulkan only ‚Äî native (non-winit) backends on Windows/macOS remain deferred to Cross.P, and a live minimized-window fallback-throttle demonstration was attempted but found infeasible on this Wayland/GNOME session (no compositor-IPC tool available to minimize a native Wayland surface); the fallback's decision logic is instead proven by a bounded-iteration unit test, mutation-verified. Per-realm pacing (`UiRealm`'s realm-scoped scheduler, ADR-0027) is out of this ADR's scope ‚Äî each realm currently drives its own frame loop through the same shared mechanism, with no cross-realm pacing coordination yet designed. Wayland freezes tickers while a surface is occluded (frame callbacks simply stop arriving ‚Äî battery-correct, no code needed); Windows/X11 minimized windows keep delivering redraws, bounded by the fallback throttle instead. Time-based UX (e.g. a `SnackBar` auto-dismiss) does not progress while a ticker is frozen this way ‚Äî the correctness fix for that is a future wall-clock Timer service, not part of this ADR.

**The `flui` facade crate + `flui::prelude` is formalized** (landed 2026-07-17): the root crate re-exports every layer an application author needs as a named module (`flui::{types, geometry, foundation, view, widgets, animation, material, cupertino, app}`, plus `flui::run_app`), and `flui::prelude` curates the everyday surface ‚Äî `flui-widgets`' own prelude plus the common Material widgets (`Theme`/`ThemeData`/`Scaffold`/`AppBar`/buttons/`Card`/`Dialog`/`SnackBar`/navigation/tabs ‚Äî see the module's own doc comment for the exact curation rule and the one deliberate omission, `TextField`, which collides in name between `flui-widgets` and `flui-material`). `flui-material`/`flui-cupertino` moved from dev-dependencies to regular dependencies of the root crate so the facade can re-export them; both are on ADR-0028's design-system-dependent allowlist already. `examples/material_demo` is migrated onto `flui::prelude`/`flui::material`/`flui::widgets`/`flui::run_app` as the ergonomics proof ‚Äî its acceptance test (`tests/material_demo.rs`) stays green unchanged, proving the facade surface suffices for a real app. A facade smoke test (`tests/facade_smoke.rs`) mounts a `flui::prelude`-only tree through the headless pipeline and exercises `flui::material`/`flui::cupertino` construction directly. **Deferred:** the other examples (`cupertino_demo`, `vertical_slice_demo`, ‚Ä¶) still import the layer crates directly ‚Äî migrating them is pure churn with no new ergonomics evidence, not a gap in the facade itself.

Needs a `flui-rendering`-scoped repro before it can be fixed or ruled out. **Known gap:** the clip family (`ClipRect`/`ClipRRect`/`ClipOval`/`ClipPath`, `crates/flui-widgets/src/clip/`) has three unwired custom-clip surfaces, surfaced by the clip parity port (`parity/clip_test.rs`, 2026-07-18): (1) `ClipRect` and `ClipOval` accept no custom clipper at all ‚Äî `RenderClip<S>` (`crates/flui-objects/src/proxy/clip.rs`) only ever resolves `S::default_for_size` for those two shapes, and `flui_rendering::delegates::CustomClipper<T>` is a fully-defined trait (with a default `RectClipper` impl) that is exported from `flui-rendering` but has zero call sites anywhere in the render-object or widget layers ‚Äî Flutter's `ClipRect(clipper: CustomClipper<Rect>)`/`ClipOval(clipper: ...)` (reclip-on-`shouldReclip` included) has no FLUI equivalent to port `clip_test.dart`'s `'ClipRect'` case onto; (2) no `ShapeBorder`/`ShapeBorderClipper`/`ClipPath::shape` constructor exists, so `ClipPath::new` only takes a bare `Fn(Size) -> Path` closure with no ambient `TextDirection` ‚Äî `clip_test.dart`'s `'ClipPath.shape'` case (identity-cached reclip across `Directionality` changes) has no equivalent; (3) `ClipRRect` only accepts a physical `BorderRadius` ‚Äî `flui_types::styling::BorderRadiusDirectional` exists as a standalone type but there is no builder path from it into `ClipRRect`, and `RenderClipRRect` carries no `text_direction` field to resolve one against even if the widget accepted it ‚Äî `clip_test.dart`'s `'ClipRRect supports BorderRadiusDirectional'` and `'ClipRRect is direction-aware'` cases have no equivalent. Separately (not a gap, a harness-fidelity bug fixed in the same pass): `flui-widgets`' headless test harness (`tests/common/mod.rs`, `LaidOut::route_event`/`dispatch_pointer_*`) hit-tested *outside* the binding's interaction-lane scope, only wrapping the dispatch half ‚Äî production (`crates/flui-app/src/app/runner.rs`, `realm.enter(|realm| event.run(realm))`) hit-tests and dispatches from inside the same lane entry. Since hit-testing can now resolve lane-registered owner-local state (`ClipPath`'s path-clip target via `resolve_path_clip_target`), the narrow scoping silently fell back to the default (whole-box) clip during any test-harness pointer dispatch ‚Äî caught by the `ClipPath` custom-clipper hit-test port and fixed by widening the lane scope to cover hit-testing too. **Known gap:** the `Transform` widget (`crates/flui-widgets/src/layout/transform.rs`) has no `transformHitTests` toggle at all, surfaced by the Transform parity port (`parity/transform_test.rs`, 2026-07-18) ‚Äî Flutter's `RenderTransform`/`Transform` (`rendering/proxy_box.dart`, `widgets/basic.dart`, both 3.44.0) carry a `transformHitTests` field (default `true`) that, when `false`, hit-tests the child ignoring the transform while `applyPaintTransform`/`localToGlobal` still honor it unconditionally; FLUI's `RenderTransform::hit_test` always inverts `effective_transform`, with no opt-out. The sibling render object in the same file, `RenderFractionalTranslation`, already carries this exact toggle (`transform_hit_tests`, exercised by that same parity port), so the gap is `Transform`-specific, not systemic to the render-object family. **Known gap:** `Transform`'s bare-matrix constructor (`Transform::new`) defaults `alignment` to `Alignment::CENTER` unconditionally, where Flutter's bare `Transform(transform:, origin:)` constructor defaults `alignment` to `null` (no contribution ‚Äî an `origin` set alone acts alone). Also surfaced by the Transform parity port: Flutter's `Transform.rotate`/`Transform.scale`/`Transform.flip` factories default `alignment` to `Alignment.center` explicitly (matching FLUI's default), and `Transform.translate` is pivot-invariant either way, so this only diverges for `origin`-only usage of the general constructor ‚Äî attempting to port `transform_test.dart`'s `'Transform origin'` case (3.44.0) found FLUI computing pivot `(150.0, 100.0)` where Flutter's null-alignment default computes `(100.0, 50.0)` for the same 100√ó100 box and `origin`, because FLUI's CENTER default adds its own `(50, 50)` contribution on top. Neither gap has a fix attempted here ‚Äî both are documented divergences from the parity port, not corrected in the same pass. **Entry:** continuous from Core.0. **Exit:** the foundation is declared stable ‚Äî all critical-tier defects closed, second-tier addressed opportunistically. This is the audit-and-repair methodology as permanent discipline, not a bounded effort.

## Governance ‚Äî how a phase becomes work

This roadmap is the index, not the work. Each phase (and each large family within a phase) is decomposed **spec ‚Üí plan ‚Üí tasks ‚Üí implement**, one directory per unit under `specs/` (name-based directories, e.g. `specs/animation-scheduling/`). Specs and plans are working artifacts, not durable authority: anything load-bearing they decide must be restated in the code, its tests, or a durable doc (`FOUNDATIONS.md`, `PORT.md`, this file) ‚Äî a reader must never need a planning doc to understand why the code is shaped the way it is. The core contracts (C2, C3, C4+C6) were the first such units and are locked, with their design documents merged into `docs/designs/`.

Every phase exit is enforced by the standing discipline of [`FOUNDATIONS.md` Part VI](FOUNDATIONS.md#part-vi--the-standing-quality-discipline): `cargo build`/`clippy`/`test` green, `bash scripts/port-check.sh` green for all refusal triggers (22 numbered plus named guards today), coverage thresholds met via `just coverage`. A phase is not done because its code is written ‚Äî it is done when its exit criteria are objectively verified.

Progress is reported as **parity against `.flutter/`** ‚Äî the scoreboard at the top of this document is the live measure. The destination is the column on the right reading ~100% in both coverage and fidelity.
