# FLUI Roadmap Tracker

> Live execution checklist for [`docs/ROADMAP.md`](ROADMAP.md). The roadmap is the *plan*; this file is the *queue* — every deliverable broken into a row with a status, an objective gate, a link to the SDD/plan artifact that owns it, and any hard sequence constraints.

**This file is the single source of truth for "what's in flight, what's done, what's next."** Update it in the `archive` phase of every SDD change, or when a Mythos receipts/plan transitions state. Do not let it drift — a stale tracker is worse than no tracker.

---

## How to use this file

- **One row = one piece of objectively verifiable work** with an exit gate copied from ROADMAP.md.
- **Status legend:**
  - `☐ todo` — not started
  - `◐ in-progress` — SDD change / plan / Mythos cycle is actively open; row links to it
  - `✓ done` — exit gate is verified green (test passes, command exits 0, file exists)
  - `⚠ verify` — believed done but unconfirmed; **first action** for a row in this state is an audit, not new work
  - `🛇 blocked` — cannot start; row lists what unblocks it
- **Owner link** points to the canonical artifact: `specs/NNN-*`, `docs/plans/YYYY-MM-DD-*`, `docs/research/YYYY-MM-DD-*`, or `openspec/changes/<name>/`.
- **Update protocol:**
  1. Moving a row to `◐ in-progress` requires linking the SDD change or plan that owns it.
  2. Moving a row to `✓ done` requires the exit gate command output or test path in the row's notes.
  3. Adding a new row to Core.0 requires a roadmap amendment — Core.0 scope is locked by ROADMAP.md.
- **Parity scoreboard** lives in [`ROADMAP.md`](ROADMAP.md#the-destination), not here. This file tracks **construction**, not parity.

(2) FLUI's `walk_strict_ancestors` is *strict*, so `Navigator::of` must re-add Flutter's self-check (`navigator.dart:2947`) itself or a nested navigator silently targets its parent. **One forced divergence, needing sign-off before any public API:** a heterogeneous `Vec<Box<dyn Route>>` cannot carry the pop result's `T`, so `pop(result)` must cross a `dyn Any` boundary and the route downcasts — Flutter has the same runtime failure mode (`Route<dynamic>` + unchecked `pop<T>`), but Rust would not otherwise need it. **One latent bug flagged, independent of this ADR:** `UNVERIFIED` whether production `build_scope` holds `WidgetsBindingInner`'s write lock across a build; if it does, `GlobalKey::current_element()` from inside a build already deadlocks (the reparent tests bypass the binding via `test_only_set_global_key_registry`). **Precondition for U1:** `crates/flui-app/src/overlay/` (568 ln, publicly exported, referenced nowhere, holds no view content, priority-sorted rather than insertion-ordered) is speculative scaffolding in H7's category and its `OverlayEntry` collides by name — delete or rename before the real one lands. **Sequence:** U1 `Overlay`/`OverlayEntry` (private) → U2 route model + lifecycle + flush + result channel (private, pure data) → U3 private `Navigator` view/element + `Navigator::of` → **U4 parity gate + `dyn Any` sign-off gate, then public export** (export follows the gate, per ADR-0017 U4 / ADR-0018 U6 precedent and the Definition of Done) → U5 `TransitionRoute`/`ModalRoute`/`PageRoute` (probably its own ADR — train-hopping is not a small port). **Deferred with named blockers:** `opaque`/`maintainState`/`skip_count` (cost: covered routes stay built/laid out/painted, `maintain_state == false` unobservable); declarative `Router`/page-based Navigator 2.0 (hence `RouteLifecycle::staging` omitted, 15 states not 16); named-route generation; restoration (no `RestorationManager` exists); **per-route focus scope (no `Focus`/`FocusScope` widget exists — H4 shipped only the `flui-interaction` manager)**; predictive back; `LocalHistoryRoute`; `OverlayPortal` (proven unnecessary — no route touches it). **U5.3 landed (ADR-0020 §7d):** `flui-objects::RenderTheater` (a `StackFit::Expand` stack whose first `skip_count` children are skipped in layout, paint and hit-test) plus the port of `OverlayState.build`'s onstage loop, so `Overlay` honours `opaque` / `maintain_state`; and a private `ModalRoute` over `TransitionRoute` (barrier + `Offstage` page merged into the single overlay entry ADR-0019 U3 keys per route). `RouteBinding` now carries the navigator's entry map, so `_handleStatusChanged`'s `overlayEntries.first.opaque` writes — unimplemented in U5.2 — are ported for all four status arms. Not claimed: per-child semantics skipping in the theater, `canSizeOverlay`, `tickerEnabled: false`, `FocusScope`, `BlockSemantics`, barrier semantics, `AnimatedModalBarrier`. Nothing new is exported. **U5.4 landed (ADR-0020 §7e):** public `PageRoute` / `PopupRoute` over the private `ModalRoute`, with `RoutePageBuilder` / `RouteTransitionsBuilder` / `RouteAnimation` aliases and the opaque `RouteBindingSlot`. `BoundRoute` + `push_bound` deleted: `NavigatorRoute::binding_slot()` lets the one public `push` bind any route without exposing `RouteBinding`. `PageRoute`'s `canTransitionTo`/`canTransitionFrom` pair became a symmetric `TransitionGroup` on the published peer, so a popup over a page drives no secondary animation. **A real deadlock fell out:** `Vsync::tick_all` held its registry lock across `tick_at`, so a route disposing from its own status listener re-entered `Vsync::unregister` and hung — fixed in flui-animation and regression-tested. Still private: `Overlay`, `OverlayEntry`, `TransitionRoute`, `ModalRoute` (Rust has no subclassing; an extensible route trait is a separate sign-off). ## Conventions

- **Adding a row.** New Core.0 deliverable requires a ROADMAP.md amendment first. Cross-track rows can be added as new D-codes (H8, P8, D5, …) without amending Core.0 scope.
- **Promoting `⚠ verify` to `✓ done`.** Run the exit-gate command (test, grep, port-check, build) and paste the verifying command output or path into the row's notes column.
- **Demoting `✓ done` back.** If a regression is found, re-open with `◐ in-progress` + a new owner link to the fix change. Never delete the row.
- **SDD change naming.** Use `core0-<id>` for Core.0 rows (e.g. `core0-n1-d1-layout-phase`), `core1-<id>` for Core.1, etc., so `openspec/changes/` filenames mirror tracker IDs.
- **Engram memory.** Save significant per-row discoveries with `topic_key: roadmap/<phase>/<id>` (e.g. `roadmap/core0/n1`) so cross-session context is recoverable.
