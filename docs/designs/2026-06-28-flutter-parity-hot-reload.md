# Flutter-Parity Hot Reload ‚Äî Architecture Design

**Date:** 2026-06-28  
**Status:** approved direction ‚Äî implementation in phases  
**Supersedes:** the "hot-restart only" conclusion in `docs/research/2026-05-22-architectural-contracts.md` Contract 9 (partially ‚Äî see ¬ß2)

---

## 1. Goal

Match Flutter's developer experience:

| Flutter | FLUI target |
|---------|-------------|
| **Hot reload** ‚Äî code changes, `State` preserved, `build()` re-run | Same semantics |
| **Hot restart** ‚Äî code changes, state reset, process kept | Same |
| **Full restart** ‚Äî kill process | `flui run` today |

This is **not** a quick-win layer-1/layer-2 cleanup. It is a **product requirement** for DX parity with Flutter, Makepad, and modern Rust UI tooling.

---

## 2. Why the current `PluginPipeline` model cannot reach parity

Today `app_plugin!` puts the **entire** three-tree inside the `.so`:

```text
Host (binary)          Plugin (.so)
  GPU, window            WidgetsBinding
  Scene pointer  ‚Üê‚îÄ‚îÄ     Element tree + ALL State
                         PipelineOwner
```

On reload the `OnceLock` resets ‚Üí **hot restart**. State lives in the plugin address space; when the `.so` is replaced, every `State<T>` is gone.

**This is the wrong split.** Research doc Contract 9 correctly identified the symptom but drew the wrong conclusion ("stateful reload is unreachable with cdylib"). The fix is **not** to abandon parity ‚Äî it is to **invert ownership**:

```text
Host (binary) ‚Äî IMMUTABLE across reload
  WidgetsBinding, BuildOwner, ElementTree
  ALL ViewState inside StatefulBehavior
  PipelineOwner, RenderTree, GPU, window

Worker (reloadable) ‚Äî SWAPPED on reload
  User `build()` implementations only
  Stateless view configs (recreated each build)
```

Flutter's Dart VM keeps the heap in the host; only **method bodies** change. FLUI's analogue: keep the **retained trees in the host binary**, reload only the **worker crate** that exports build logic.

Makepad achieves parity via a **Live VM / Splash** (interpreted layer) for declarative UI ‚Äî a valid Phase 3 option for FLUI widget *definitions*, but Phase 1 targets native Rust via host/worker split (closer to Flutter's mental model for widget authors).

## 3. Three tiers (Flutter vocabulary)

```rust
pub enum HotReloadTier {
    /// Re-run `build()` on the whole element tree; preserve `State`.
    /// Flutter: `flutter run` + save ‚Üí hot reload.
    HotReload,

    /// Remount root widget; drop all `State`; keep process + GPU context.
    /// Flutter: hot restart (`R` in CLI).
    HotRestart,

    /// Kill process; `flui run` respawn.
    FullRestart,
}
```

Mapping to dev orchestration ([`ReloadStrategy`](../hot-reload.md)):

| HotReloadTier | Requires | ReloadStrategy (build side) |
|---------------|----------|----------------------------|
| HotReload | Host/worker split + successful code swap | Build worker dylib only |
| HotRestart | Host remount API | Build worker dylib or full app |
| FullRestart | ‚Äî | ProcessRestart |

## 4. Hot reload protocol (Flutter `reassemble` port)

On **HotReload** after the worker dylib is rebuilt and reloaded:

1. **Invalidate const caches** (Phase 2 ‚Äî `const` view memoization).
2. **`BuildOwner::reassemble(tree)`** ‚Äî mark every element dirty (Flutter `Element.reassemble`).
3. **`PipelineOwner::reassemble()`** ‚Äî call `RenderObject::reassemble`, mark layout+paint dirty.
4. **`WidgetsBinding::draw_frame()`** ‚Äî drain build scope; reconciliation runs with **preserved** elements.
5. **`RendererBinding::draw_frame()`** ‚Äî layout/paint as usual.

State preservation is automatic if step 4 never unmounts stateful elements (same `ElementId`, same `can_update`).

On **layout/type change** that breaks `can_update` ‚Üí degrade to **HotRestart** (explicit, logged ‚Äî Flutter does the same with reload rejection).

## 5. Project structure (worker/host/types)

Every hot-reloadable app uses a **three-crate** layout:

```text
my_app/
  my_app_types/     # rlib ‚Äî State structs, view configs (stable layout)
  my_app_logic/     # cdylib ‚Äî build fns, widget impls (reload target)
  my_app_host/      # bin ‚Äî thin main, links flui-app + loads logic
```

| Crate | Linked into | Contains |
|-------|-------------|----------|
| `my_app_types` | host + logic | `CounterState`, shared enums, `Serialize` snapshots (Phase 2) |
| `my_app_logic` | cdylib only | `impl ViewState::build`, widget tree |
| `my_app_host` | binary | `run_app`, `HotReloadEngine`, trees |

**Rule:** `ViewState` memory lives in the host `Element` tree. The worker never owns long-lived state ‚Äî only constructs fresh `View` configs.

Worker exports (evolution of `app_plugin!`):

```rust
// Generated by hot_reload_app! macro
extern "C" fn flui_worker_version() -> u32;
extern "C" fn flui_worker_build_root(ctx: *mut HostBuildContext) -> *mut c_void; // Box<dyn View>
extern "C" fn flui_worker_type_fingerprint() -> u64; // for layout-change detection
```

`HostBuildContext` is an opaque handle to the host's `BuildContext` / element owner ‚Äî passed **into** the worker each build, never stored in the worker.

## 6. Code reload mechanisms (Phase 1 vs 2)

### Phase 1 ‚Äî Worker dylib swap (MVP parity path)

- Host loads `my_app_logic.dll` via existing `DynLib`.
- On mtime change: `unload` ‚Üí `load` ‚Üí `perform_hot_reload(HotReloadTier::HotReload)`.
- Worker rebuilt by `cargo watch -p my_app_logic` or `flui run` orchestration.
- **Constraint:** `my_app_types` must not change layout across reload for HotReload tier; fingerprint mismatch ‚Üí HotRestart.

### Phase 2 ‚Äî Function-level reload (`hot-lib-reloader`)

- Worker functions annotated with `#[hot_module]` for faster swap without full dylib churn.
- `before_reload` / `after_reload` hooks for opt-in state serialization.
- Used when worker crate is large and link time dominates.

### Phase 3 ‚Äî Live layer (Makepad Splash / Rhai)

- Declarative widget trees in hot-swappable scripts for designer workflow.
- Rust worker for performance-critical paths.
- Optional; not required for widget-author parity.

## 8. Non-goals (honest limits)

- **Arbitrary `State` layout changes across HotReload** ‚Äî same as Flutter (reload fails ‚Üí restart).
- **Global statics in worker** ‚Äî leak or stale; forbidden by worker crate lint.
- **`tracing` subscriber in worker** ‚Äî known `hot-lib-reloader` hazard; host owns tracing.
- **Production hot reload** ‚Äî dev-only; release builds static-link worker.
