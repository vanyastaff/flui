# ADR-0015: Resolved-position hit-testing for `RenderFollowerLayer` rides a `RenderId`-keyed side table of composite-resolved offsets on `PipelineOwner` — produced by a small post-paint step in flui-rendering (reusing the already-landed `resolve_follower_offset`), read generically by the existing hit-test walk — NOT a `hit_test_transform` signature change and NOT a live per-event re-resolution

*A `CompositedTransformFollower` paints at a position resolved at composite time against the fully-built `LayerTree` (`flui_layer::resolve_follower_offset`, called from flui-engine's `render_layer_recursive`), but FLUI's hit-test is a **separate walk over the render tree** (`PipelineOwner::hit_test`) with **zero coupling** to the layer tree, the `LinkRegistry`, or any frame state — so a tooltip/dropdown that has floated away from its natural tree position hit-tests at the WRONG on-screen location. The fix is **one channel**: capture the follower's `RenderId → LayerId` correlation as a near-free byproduct of the paint walk (the general reusable primitive multiple pending consumers already want), resolve each follower's offset **once, post-paint, inside flui-rendering** using the same `resolve_follower_offset` the GPU path already uses, and stash the results in a `RenderId → Offset` side table on `PipelineOwner` (`last_follower_offsets`) that mirrors the existing per-frame byproducts (`last_layer_tree`, `last_link_registry`, `pending_child_requests`). The hit-test walk reads that side table **generically** — riding the identical `push_transform`/`pop_transform` lifecycle and position-shift pattern the walk already uses — leaving `RenderFollowerLayer::hit_test` a plain structural forward and leaving `hit_test_transform`'s signature untouched. This is Flutter's own `getLastTransform()` cache-from-last-composite contract, ported faithfully: hit-test consults the LAST completed frame's resolved position, never a live recompute.*

## Context

### What `RenderFollowerLayer` hit-testing needs (oracle: `proxy_box.dart` + `layer.dart`)

Flutter's `RenderFollowerLayer` does **not** re-derive its transform during hit-testing. It reads a value **cached on the compositor layer from the last completed compositing pass**:

- `getCurrentTransform() => layer?.getLastTransform() ?? Matrix4.identity()` (`proxy_box.dart:4668-4670`). `FollowerLayer._lastTransform` is written by `_establishTransform()` during `addToScene` (`layer.dart:2797-2841`), the composite pass that runs strictly **after** the whole paint phase; `getLastTransform()` just returns that cached matrix (`layer.dart:2668-2680,2707-2714`).
- `hitTest` (`proxy_box.dart:4672-4683`): `if (link.leader == null && !showWhenUnlinked) return false;` then delegates to `hitTestChildren` — the follower **never** adds itself as a hit target, only forwards.
- `hitTestChildren` (`proxy_box.dart:4685-4694`): wraps the default child forward in `result.addWithPaintTransform(transform: getCurrentTransform(), ...)` — i.e. hit-testing re-applies the **same cached transform** paint used, consistent-by-construction with what is on screen.

The load-bearing property (pinned down precisely, not hand-waved): **pointer events are dispatched between frames**, after a complete frame's paint+composite already ran, so in steady state `getLastTransform()` is exactly up to date; the only staleness window is mid-frame, which is unobservable to hit-testing in Flutter's single-threaded UI model. Consulting the last completed composite's resolved position is **the correct, intended behavior — not a shortcut**. Any FLUI design must replicate *that* contract, not try to be "more live" than the oracle.

### The verified current state of FLUI (file:line, not assumption)

**1. The hit-test walk is a render-tree-only walk with zero layer-tree coupling.** `PipelineOwner::hit_test(&self, position, result)` (`crates/flui-rendering/src/pipeline/owner/accessors.rs:314`) → `hit_test_subtree` → `hit_test_subtree_impl` (`:342`). It reads `render_tree` exclusively; the node's `own_size` comes from `entry.state().geometry()` (`:366`); there is **no** reference to `LayerTree`, `Scene`, or `LinkRegistry` anywhere in the walk. It is `&self` and can run at arbitrary times relative to a paint/render pass.

**2. `hit_test_transform` takes zero external context.** `RenderObject::hit_test_transform(&self, size) -> Option<Matrix4>` (`crates/flui-rendering/src/traits/render_object.rs:487`), forwarded from `RenderBox` (`render_box.rs:408,711`) and `RenderSliver` (`render_sliver.rs:391,603`). The walk calls it at `accessors.rs:374`, pushing the result onto the `HitTestResult` transform stack (`push_transform`, `:377`) before recursing and popping it after the parent entry is added (`pop_transform`, `:446`). Its **only two real overriders** — `RenderTransform` (`crates/flui-objects/src/layout/transform.rs:240`) and `RenderRotatedBox` (`rotated_box.rs:195`) — are **self-contained**: each derives its matrix purely from its own laid-out `size`/rotation, needing no cross-tree data. The hook is *deliberately* context-free.

**3. The walk's transform hook only feeds the result stack; the object's `hit_test` body shifts child positions.** For `RenderTransform`, the object inverts the matrix itself and forwards the inverse-mapped point to children (`transform.rs:212-229`), while `hit_test_transform` returns the *forward* matrix for the result stack (`transform.rs:234-242`; the comment at `:225-228` documents this split). So the two concerns — (a) the position handed to children, owned by the object's `hit_test` body; (b) the accumulated transform on the result stack for later event dispatch, fed by `hit_test_transform` — are separate and must be kept consistent by the implementor.

**4. `RenderFollowerLayer::hit_test` today is the plain structural forward.** `crates/flui-objects/src/proxy/follower.rs:268-284`: `if !self.has_child { return false; } ctx.hit_test_child_at_offset(0, Offset::ZERO)`. It forwards the hit at the follower's own **layout-relative** offset, with an explicit module-doc deferral (`:27-31`) stating resolved-transform hit-testing is "a genuine chief-architect ADR question, not a shortcut to invent in this pass." `hit_test_child_at_offset(index, offset)` subtracts the offset from position before recursing (`context/hit_test.rs:305-308`, `position_minus`). This is exactly the "wrong on-screen location" problem: the follower forwards at its tree position, ignoring the composite-time displacement.

**5. Tier-2 render-time resolution has fully landed:** `flui_layer::resolve_follower_offset(tree, registry, follower_layer_id, follower)` (`crates/flui-layer/src/link_registry.rs:330`) walks both ancestor chains in the fully-built `LayerTree` to their common ancestor, sums the `Layer::Offset` deltas, and feeds `FollowerLayer::calculate_offset`. It returns `Option<Offset>` — **`None` means "do not render this subtree at all"** (unlinked AND `show_when_unlinked == false`, `:335-347,761-775`). flui-engine's `render_layer_recursive` calls it (`crates/flui-engine/src/wgpu/renderer.rs:1578-1604`), and the binding threads the per-frame registry through `Scene::with_links` via `owner.take_link_registry()` (`crates/flui-app/src/app/binding.rs:699,747`). **The pixel path is correct; only hit-test is stale.**

**6. No `RenderId → LayerId` correlation exists anywhere — confirmed, and independently wished-for.** The only hits across the workspace are TODO-shaped comments explicitly noting its absence: `crates/flui-rendering/src/testing/snapshot.rs:906-910,955` ("No RenderId→LayerId mapping exists yet; fall back to the whole tree") and `crates/flui-rendering/src/testing/harness.rs:568,774` ("Falls back to the full tree until a `RenderId → LayerId` mapping is [available]"). `LinkRegistry` (flui-layer) is deliberately `RenderId`-agnostic — flui-layer is a foundation crate with no `flui-rendering` dependency, and correctly should not know render-tree identities.

**7. The precedent for "render-time-computed, hit-test-time-read" per-node data already exists — it is `RenderNode::offset()`.** The hit-test walk reads a child's committed layout offset at `accessors.rs:400,414` (`child_node.offset()`), the paint walk reads the same at `paint.rs:306`, and both are the value the *layout* phase committed onto `RenderState`. A follower's composite-resolved offset is **the identical shape of data** — computed in a pipeline phase, cached per node, read later by the walk — just resolved one phase further along (after paint/composite, not after layout). This ADR does not invent a new *kind* of channel; it adds a second, phase-appropriate instance of one FLUI already relies on.

**8. `PipelineOwner` already retains what the resolution needs, and flui-rendering can already run it.** The owner holds `last_layer_tree: Option<LayerTree>` (`pipeline/owner/mod.rs:156`) and `last_link_registry: Option<LinkRegistry>` (`:163`), both carried across the typestate phases by `rebind_phase` (`:267-268`). `take_link_registry()` `.take()`s the registry out to the Scene (`accessors.rs:235-236`). Critically for the crate boundary: **flui-rendering depends on flui-layer** (`Cargo.toml:20`) and can call `resolve_follower_offset` directly; it does **not** depend on flui-engine (`Cargo.toml:15-21` — types/painting/interaction/foundation/tree/layer/semantics, no engine). flui-engine depends on flui-layer *and* flui-rendering, so any attempt to have the GPU pass write resolved values back into the owner would be an **upstream write from a downstream crate** — the wrong direction.

### What is genuinely missing (the whole decision)

Exactly one channel: **a way to carry each follower's composite-resolved offset from the point it becomes resolvable (after the paint walk completes the `LayerTree`) to the point the hit-test walk visits that follower's `RenderId`.** Everything else — the resolution math (`resolve_follower_offset`), the retained tree+registry, the transform-stack lifecycle, the position-shift pattern, the structural-forward body — already exists. The one blocker is that the resolution is keyed by `LayerId` while the hit-test walk speaks `RenderId`, and no correlation between the two exists.

## Decision

**Introduce one channel: a `RenderId`-keyed side table of composite-resolved follower offsets, produced by a small post-paint step in flui-rendering (reusing `flui_layer::resolve_follower_offset`), and read generically by the existing hit-test walk.** Add **no** crate dependency, **no** lock in public API, **no** async on a hot path, **no** change to `hit_test_transform`'s signature, and **no** restructuring of the walk's recursive shape.

### D1 — The carrier: `last_follower_offsets: FxHashMap<RenderId, Offset>` on `PipelineOwner`, a per-frame byproduct like `last_layer_tree`/`last_link_registry`

Add one field to `PipelineOwner` (`pipeline/owner/mod.rs`, threaded through `rebind_phase`), mirroring the existing paint-phase byproducts. Semantics of an entry:

- **Present, `Some(r)` → hittable at offset `r`:** the follower's subtree should be hit-tested as if translated by `r` (the same `r` the GPU path pushes via `backend.push_offset(resolved)`, `renderer.rs:1586`).
- **Absent → nothing to do:** the node is either not a follower, or a follower that resolved to `None` this frame (unlinked + `show_when_unlinked == false`). For the follower case, absence is **load-bearing**: the walk skips the subtree, exactly mirroring the render path's `resolve_follower_offset → None → don't descend` (`renderer.rs:1579-1603`) and oracle's early return in `FollowerLayer.addToScene` (`layer.dart:2857-2865`).

The map is **empty for the overwhelming majority of trees** (followers are rare — tooltips, dropdowns, overlays), so the walk gates all new work behind a single `if !self.last_follower_offsets.is_empty()` check. It is a private field accessed only through methods — never returned as a guard — so SP-6 / the "no locks in public API" port-check is respected by construction.

### D2 — The correlation: `FragmentComposer` records `RenderId → LayerId` for each `Layer::Follower` it pushes (the general primitive, narrowed to followers now)

The `FragmentComposer` (`paint.rs:344-355`) already builds the `LayerTree` and populates the `LinkRegistry` as a byproduct of `push_layer` inserting each `Layer::Follower`. At that push site the composer knows the `LayerId` it just inserted, and the calling `paint_subtree_impl` already has the render node's `node_id` in scope (`paint.rs:157-160`, threaded into every fragment-op replay including `FragmentOp::Push`, `:272`). Recording `(node_id, follower_layer_id)` there is a **near-free, mechanical addition** — no new walk, no new pass.

This correlation is the **general reusable primitive** the snapshot/harness subtree-scoping TODOs (Context §6) independently want. Per ADR-0014's scope discipline (general `excludes_semantics_subtree` vs. narrow), this ADR **introduces the correlation justified by its multiple pending consumers but ships only the follower subset needed now** — a `Vec<(RenderId, LayerId)>` (or equivalent) of the frame's followers, not yet the full every-node map. The full map is a clean additive superset when a snapshot/harness consumer actually lands. The correlation lives **flui-rendering-side** (on the owner / composer); flui-layer's `LinkRegistry` stays `RenderId`-agnostic — the crate boundary is preserved.

### D3 — The resolution: one post-paint step in flui-rendering, reusing the shared `resolve_follower_offset`

After the paint walk completes and `composer.finish()` yields the **fully-built** `LayerTree` + `LinkRegistry` (the point at which every leader anywhere in the tree is already composed, so resolution is order-independent — the same same-frame guarantee the GPU path relies on), iterate the captured follower correlations and call, for each:

```
flui_layer::resolve_follower_offset(&tree, &registry, follower_layer_id, follower)
```

`Some(r)` → insert `(render_id, r)` into `last_follower_offsets`; `None` → omit (hidden this frame). This runs **before** `take_link_registry()` hands the registry to the Scene (or against the retained fields), entirely within flui-rendering using the flui-layer function it already depends on. It is the **same algorithm the GPU path uses** — one function, two consumers (pixels + hit-test), **not** two copies of the logic. It is a sync, between-phase step (naturally placed at the tail of the paint phase / head of the next), so it touches no layout/paint/hit-test async port-check trigger.

*Honest note on the double computation:* Flutter resolves once (`_establishTransform`) and both pixels and hit-test read the single cached `_lastTransform`. FLUI computes the same numbers twice — once in flui-engine for pixels, once in flui-rendering for hit-test — because the two consumers live in different crates and a single shared computation would require the downstream GPU crate to write back into the upstream owner (D-rejected in Alternatives). The cost is trivial (a handful of followers, a translation-only sum over a short ancestor chain), and the *logic* is single-sourced in `resolve_follower_offset`. Unifying the two call sites into one computation whose result the engine reads is a possible future cleanup, explicitly **out of scope** (D6) — this ADR keeps the already-landed render path untouched.

### D4 — The consumption: the hit-test walk reads the side table generically, riding the existing transform lifecycle

In `hit_test_subtree_impl` (`accessors.rs:342`), before recursing into a node's subtree, consult `last_follower_offsets` for the current `RenderId`. When a resolved offset `r` is present, the walk applies it as a hit-test transform sourced from the side table instead of from the object's `hit_test_transform(size)` hook — reusing the **identical** machinery already in the walk:

- Push `Matrix4::translation(r)` onto the `HitTestResult` transform stack (`result.push_transform`, the same call at `accessors.rs:377`), popped after the node's entry is added (the same `pop_transform` at `:446`) — so captured entries carry the correct accumulated transform for later event dispatch, exactly as `hit_test_transform` results do.
- Shift the position handed into the follower's subtree by `-r` (the inverse of the paint translation), reusing the existing per-recursion position-shift pattern the walk already uses for child offsets (`result.with_paint_offset(child_offset, |result| ... position - child_offset ...)`, `accessors.rs:401,415`). This is what makes children hit-test at the leader-relative on-screen location.

Because the position shift is applied by the walk, `RenderFollowerLayer::hit_test` **stays the plain structural forward** — its body needs no change beyond honoring the "hidden → subtree skipped" case, which the walk already enforces via the side-table absence (D1). `hit_test_transform`'s signature is **untouched**; no existing implementor is affected. `HitTestResult` is the canonical `flui_interaction::routing::HitTestResult` (re-exported); its `push_transform`/`pop_transform`/`with_paint_offset` primitives (`flui-interaction/src/routing/hit_test.rs:303,317,357`) are the same ones the walk already drives. The walk's fundamental recursive shape is unchanged — one gated side-table lookup per node, folded into the existing transform-stack lifecycle.

### D5 — Loyalty: this is `getLastTransform()`, ported faithfully — cache-from-last-composite, not live re-resolution, translation-only by existing FLUI scope-down

The side table **is** FLUI's `_lastTransform`: resolved once per frame after the composite is knowable, read by every hit-test until the next frame replaces it. The one-frame-staleness window (a pointer event arriving after a leader moved but before the next frame's paint) is **identical to Flutter's and is the intended behavior** — do not apologize for it, do not try to make it "more correct" than the oracle by re-resolving live. The resolved value is a **translation `Offset`**, not a general `Matrix4`, matching the already-landed render path (`resolve_follower_offset` returns `Offset`) — a pre-existing, documented FLUI scope-down (FLUI's paint composer only ever shifts coordinate space via `Layer::Offset`, `link_registry.rs:396-415`), **not** a new limitation introduced here. Lifting an `Offset` to `Matrix4::translation` at the push site is exact and lossless.

### D6 — Scope: general primitive vs. follower-specific carrier (the ADR-0014 distinction, applied)

- **General (introduce, justified):** the `RenderId → LayerId` correlation (D2). It has independent pending consumers (snapshot/harness subtree scoping, Context §6) and is not speculative. Shipped narrowed to followers; the full every-node map is a clean additive extension.
- **Follower-specific (keep narrow, do NOT generalize):** the resolved-offset side table (D1) and its consumption (D4). **No other render object needs a composite-resolved hit-test position** — every other hit-test transform (`RenderTransform`, `RenderRotatedBox`, sliver variants) is self-contained from the object's own layout, resolvable inside `hit_test_transform(size)`. Building a speculative "generic render-time-resolved hit-test transform framework" would over-encode a contract exactly one object needs. The side table stays a follower carrier.

### D7 — Explicitly OUT of scope (state honestly; do not imply "next")

- **The GPU render path** (`renderer.rs:1578-1604`, `resolve_follower_offset` consumption, `Scene::with_links` wiring) — already landed and correct; **unchanged** by this ADR.
- **Unifying the two resolution call sites** into a single computation shared across pixels + hit-test — a possible future cleanup (D3 note), not needed for correctness, not attempted here.
- **`RenderLeaderLayer` hit-testing** — needs **nothing**: it is hit-test-transparent (plain forward, no override), matching oracle's reliance on `RenderProxyBoxMixin` defaults (`proxy_box.dart:4475-4535` has no `hitTest`/`applyPaintTransform`). No leader appears in the side table.
- **Full `Matrix4` follower transforms** for hit-test — translation-only, parity with the render path (D5); Flutter's general matrix is a scope-down FLUI already took.
- **Follower semantics geometry** — no semantics consumer exists for it; deferred consistent with every other catalog entry and ADR-0014's own deferrals.
- **Any other catalog item** — the catalog is complete (74/74); nothing else depends on this. This ADR closes purely `RenderFollowerLayer`'s one remaining rough edge.

## Consequences

**Positive**

- Closes the last deferred rough edge by **wiring existing machinery end-to-end**: `resolve_follower_offset` (reused, not rebuilt), the retained `last_layer_tree`/`last_link_registry`, the `FragmentComposer` byproduct pattern, and the `push_transform`/`pop_transform`/`with_paint_offset` hit-test lifecycle are all *reused*. Net new surface: one `PipelineOwner` field, one composer byproduct, one post-paint resolution loop, and one gated side-table read in the walk.
- No new crate dependency (flui-rendering already reaches `resolve_follower_offset` via flui-layer), no lock in public API (private field, method access only), no async hot path (resolution is a sync between-phase step), and **no change to `hit_test_transform`** — every existing implementor (`RenderTransform`, `RenderRotatedBox`, slivers) is untouched.
- The hit-test walk stays **decoupled from the layer tree**: it consults a pure render-tree-keyed side table, never the `LayerTree`/`LinkRegistry`. The render-tree-only shape of the walk is preserved; the layer tree is not threaded in.
- Faithful to Flutter's `getLastTransform()` cache-from-last-composite contract, with an identical staleness window — a real oracle behavior, ported, not a shortcut.
- The `RenderId → LayerId` correlation lands as a genuinely reusable primitive with named pending consumers (snapshot/harness), so the follower is its first consumer rather than a bespoke hack.
- One fact, one place: the resolved offset lives once (the side table); the resolution *logic* lives once (`resolve_follower_offset`); geometry is not duplicated onto `RenderState`.

**Negative / Trade-offs**

- **Double resolution** (flui-engine for pixels, flui-rendering for hit-test) — accepted (D3): trivial cost, single-sourced logic, avoids an upstream write from the downstream GPU crate. Named as a future unification, not a correctness debt.
- **A new per-frame allocation** (`last_follower_offsets`) — empty and effectively free for the common (no-follower) tree; a small map when followers exist, mirroring the existing per-frame byproducts.
- **One-frame staleness** for hit-testing a follower whose leader moved since the last composite — this is Flutter's exact, intended behavior (D5), not a regression; it must be **reported as the oracle contract**, never as "not yet fully correct."
- The DEV task must add the position-shift + transform-push at the follower node and honor the hidden-subtree skip; the harness proof (below) is the DoD guard that this is real, not a stub returning the tree-relative forward.

**Follow-ups (named, sequenced, not in this ADR)**

- DEV: `FragmentComposer` follower correlation byproduct + `last_follower_offsets` field + post-paint resolution loop (Slice A/B).
- DEV: hit-test walk reads the side table (position shift + transform push + hidden skip) + `RenderFollowerLayer::hit_test` honoring the skip (Slice C).
- Deferred: generalize the follower correlation to the full `RenderId → LayerId` map when the snapshot/harness subtree-scoping consumers land; unify the two `resolve_follower_offset` call sites; follower semantics geometry.

## Alternatives Considered

| Option | Why rejected |
|---|---|
| **Change `hit_test_transform(&self, size)` to take `LayerTree` + `LinkRegistry` context** so a follower can resolve itself in the hook. | Couples a widely-implemented trait method (`RenderTransform`, `RenderRotatedBox`, sliver variants, `render_object.rs:487` + four forwards) and the **entire** hit-test walk to the layer tree, for a two-object feature. The hook is deliberately context-free (self-contained overriders); threading frame state through it inverts the walk's render-tree-only shape. The side table delivers the same value keyed by `RenderId` without touching the hook or the walk's coupling. |
| **Cache the resolved offset on `RenderState`** (a per-node field alongside `offset()`), read by the walk like `offset()`. | Grows the hot per-node struct for a rare feature (a handful of followers per app), and requires a `&mut` mutation pass over the tree post-paint. It also conflates two different notions of "position" on the geometry owner: layout-committed vs. composite-resolved. The owner side table is the FLUI-native "per-frame byproduct" shape (matching `last_layer_tree`/`last_link_registry`/`pending_child_requests`), stays empty in the common case, and keeps `RenderState` a single-notion geometry owner. |
| **Resolve lazily inside the hit-test walk** — thread `last_layer_tree` + `last_link_registry` + the `RenderId → LayerId` map into the walk and call `resolve_follower_offset` per pointer event. | Re-does the ancestor-chain sum on **every** pointer event, precisely what Flutter avoids by caching `_lastTransform` (`layer.dart:2668-2714`). It also couples the render-tree walk to the layer tree + registry (currently it touches neither) and forces the owner to retain both structures solely for hit-test. Pre-resolving into a `RenderId`-keyed side table is both **more loyal** (cache-from-last-composite) and **less coupled** (the walk sees only a render-tree-keyed map). |
| **Self-cache on the render object via `Cell<Offset>` written during `paint`.** | Plan §7.5 / rejected: paint is `&self` by design, and even with interior mutability the value cached during *this* node's own paint visit is only as fresh as the single-pass ordering allows — silently wrong whenever the follower paints before its leader in the same descent (the common cross-`Overlay` case). The object also cannot see the *completed* `LayerTree` from inside its own paint, so it literally cannot compute the resolved value there. Not more correct than today's structural forward, just more likely to be mistaken for correct. |
| **Have flui-engine write resolved offsets back into the `PipelineOwner`** after its GPU resolution, so the value is computed once. | An **upstream write from a downstream crate** (flui-engine → flui-rendering owner): wrong dependency direction, couples the GPU pass to pipeline internals, and the engine pass may run detached from the owner (its own `Scene` arc, potentially another thread). Resolving independently in flui-rendering with the **shared** `resolve_follower_offset` keeps the boundary one-directional at the cost of a trivial recomputation. |
| **Fake the harness green** — special-case the follower test input, or assert the tree-relative forward as if it were resolved. | DoD anti-cheating violation: it would report "follower hit-testing done" while a displaced tooltip still hit-tests at its tree position. The gap is the resolved-position channel, not the test; papering it over is the "MVP reported as parity" failure mode AGENTS.md forbids. |

## Ordered implementation plan

All changes are confined to **flui-rendering** (the owner field + composer byproduct + post-paint resolution + walk read) and a one-line honoring of the hidden-skip in **flui-objects** (`RenderFollowerLayer::hit_test`). No workspace-manifest edits; no flui-layer/flui-engine changes.

**Slice A — the correlation byproduct (independent, landable first):**
1. `FragmentComposer` records `(RenderId, LayerId)` when it pushes a `Layer::Follower` (`paint.rs` push site, `node_id` already in scope). Expose it as a paint-phase byproduct alongside the existing `link_registry`. Unit test: painting a follower node yields one correlation entry whose `LayerId` is the pushed `Layer::Follower`.

**Slice B — the resolution + carrier (depends on A):**
2. Add `last_follower_offsets: FxHashMap<RenderId, Offset>` to `PipelineOwner` (+ `rebind_phase` threading + a private-method accessor). ## Maintainer-grade pre-code gate

**Verdict: ACCEPTABLE.** The design reuses the capabilities that already exist — `flui_layer::resolve_follower_offset` (the single-sourced resolution the GPU path already uses), the owner's retained `last_layer_tree`/`last_link_registry`, the `FragmentComposer` byproduct pattern that already builds the `LinkRegistry`, and the `HitTestResult` `push_transform`/`pop_transform`/`with_paint_offset` lifecycle the walk already drives — rather than inventing a parallel hit-test-transform subsystem or a live re-resolver. It adds **no** new crate dependency (flui-rendering already reaches flui-layer; it is deliberately kept off flui-engine to preserve the one-directional boundary), **no** lock in public API (the side table is a private field behind methods, SP-6 honored), **no** async on the layout/paint/hit-test hot path (resolution is a sync between-phase step), and **no** change to `hit_test_transform` (every existing implementor is untouched). Boundaries stay acyclic and one-directional: the resolution *logic* stays in flui-layer, the resolved *cache* and the `RenderId` correlation stay in flui-rendering where the render tree lives, and flui-layer's `LinkRegistry` stays `RenderId`-agnostic — the GPU crate is never made to write upstream. One fact, one place: the resolved offset lives once (the side table), the resolution algorithm lives once (`resolve_follower_offset`, two consumers not two copies), and geometry is not duplicated onto `RenderState`. Boundary-type check: the carrier is a plain `RenderId`-keyed side table matching the established per-frame-byproduct fields (`last_layer_tree`/`last_link_registry`/`pending_child_requests`), read through the existing transform lifecycle — deliberately smaller than a `hit_test_transform`-context change or a per-node `RenderState` field, both of which would over-couple a two-object feature. Scope discipline (ADR-0014 pattern): the `RenderId → LayerId` correlation is introduced as the general reusable primitive (justified by the independent snapshot/harness pending consumers), narrowed to followers now; the resolved-offset carrier is kept follower-specific because no other object needs a composite-resolved hit-test position. Forward view (2 years / 3 extensions): the correlation generalizes to the full `RenderId → LayerId` map for snapshot/harness subtree scoping; the two `resolve_follower_offset` call sites can later unify into a single shared computation; and any future composite-resolved hit-test consumer reuses the side-table shape — each an additive layer over a correct core, none requiring the core reshaped. The `.flutter/flutter-master/` oracle (`proxy_box.dart:4668-4694`, `layer.dart:2668-2714`) is available for the Slice-C DoD cross-check; this ADR's own milestone is the harness proof that a **visually displaced** follower is hit at its **on-screen** position. The one honesty flag the DEV task must carry: this is Flutter's **cache-from-last-composite** contract (one-frame staleness identical to the oracle), **not** a live recompute — report it as the intended behavior, never as an incomplete approximation.
